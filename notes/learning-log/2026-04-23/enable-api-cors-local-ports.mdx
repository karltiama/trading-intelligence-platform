## Context

The frontend runs on port `3000` by default; Next may use `3001` if `3000` is busy. The API listens on `3001` by default. Browser requests from the web app to the API need explicit CORS allowlist origins.

## Goal

Configure Nest `enableCors` with explicit local origins for both common dev ports (`3000` and `3001`) on `localhost` and `127.0.0.1`.

## Files Changed

- `apps/api/src/main.ts`
- `notes/learning-log/2026-04-23/enable-api-cors-local-ports.mdx`

## Implementation Notes (key logic explained)

- `app.enableCors({ origin: [...], methods: [...], credentials: true })` runs before `listen`.
- Origins are browser origins (scheme + host + port), not the API bind port.

## Key Decisions (and alternatives considered)

- **Explicit allowlist** instead of `origin: true` for clearer security posture.
- **Both `localhost` and `127.0.0.1`** because browsers treat them as different origins.

## Assumptions

- Local dev only for this list; production domains will move to env-driven config later.

## Failure Modes

- New dev host/port requires updating the list or switching to env (`CORS_ORIGINS`).

## How to Test (commands + steps)

```bash
cd apps/api
npm run lint
npm run start
```

Preflight check (PowerShell):

```powershell
Invoke-WebRequest -Uri "http://localhost:3001/health" -Headers @{ Origin = "http://localhost:3000" } -UseBasicParsing
```

Inspect response headers for `Access-Control-Allow-Origin`.

## Next Steps

- Add `CORS_ORIGINS` (comma-separated) in `.env.example` when you add deployment.
