## Context

The API scaffold was running but lacked a dedicated operational health endpoint. We needed a lightweight, production-minded check route that can be used by local dev, CI smoke checks, and future platform probes.

## Goal

Add a simple `GET /health` endpoint in `apps/api` using standard NestJS module/controller/service layering.

## Files Changed

- `apps/api/src/health/health.service.ts`
- `apps/api/src/health/health.controller.ts`
- `apps/api/src/health/health.module.ts`
- `apps/api/src/app.module.ts`
- `notes/learning-log/2026-04-23/add-health-endpoint.mdx`

## Implementation Notes (key logic explained)

- Added `HealthService` returning a minimal JSON payload:
  - `status: "ok"`
  - `service: "api"`
  - `timestamp: ISO8601 string`
- Added `HealthController` with route prefix `health` and `@Get()` handler.
- Added `HealthModule` to encapsulate health controller/service.
- Imported `HealthModule` in `AppModule` to expose `/health` endpoint.
- Kept implementation intentionally small and dependency-free.

## Key Decisions (and alternatives considered)

- **Decision:** Return static operational fields + timestamp.
  - **Why:** Fast and reliable liveness/readiness signal without external dependency checks.
  - **Alternative:** Add DB/broker dependency probes immediately (rejected for MVP simplicity and startup speed).
- **Decision:** Use dedicated module (`health`) instead of putting route in `AppController`.
  - **Why:** Clean standard Nest structure and future extensibility.
  - **Alternative:** Add `/health` in `AppController` (rejected as less maintainable as app grows).

## Assumptions

- Current need is process-level health, not deep dependency readiness.
- Port defaults are already adjusted so API can run locally on `3001` when frontend is on `3000`.

## Failure Modes

- If the app process is up but dependencies are down, `/health` still returns ok (by design).
- If API port changes unexpectedly, probe target URL must be updated.

## How to Test (commands + steps)
