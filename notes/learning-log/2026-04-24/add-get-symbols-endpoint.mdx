## Context

After Neon + Prisma setup, the next practical milestone was exposing a read endpoint so the frontend can fetch real database-backed data.

## Goal

Add `GET /symbols` to return rows from the `Symbol` table using Prisma with minimal, maintainable NestJS wiring.

## Files Changed

- `apps/api/src/app.service.ts`
- `apps/api/src/app.controller.ts`
- `apps/api/src/app.controller.spec.ts`
- `apps/api/src/main.ts`
- `notes/learning-log/2026-04-24/add-get-symbols-endpoint.mdx`

## Implementation Notes (key logic explained)

- `AppService` now injects `PrismaService` and exposes `listSymbols()`.
- `listSymbols()` uses `prisma.symbol.findMany` with ordered output and explicit selected fields.
- `AppController` adds `@Get('symbols')` that delegates to `AppService.listSymbols()`.
- Unit test was updated with a minimal `PrismaService` mock because `AppService` now has DI dependency.
- Added `import 'dotenv/config';` in `main.ts` so runtime reads `apps/api/.env` and connects to Neon instead of fallback/local DB.

## Key Decisions (and alternatives considered)

- **Decision:** Put endpoint in existing app controller/service for smallest diff.
  - **Alternative:** create a dedicated `symbols` module immediately (deferred to keep Phase 0 momentum).
- **Decision:** Explicit field selection in Prisma query.
  - **Alternative:** return full rows (rejected to keep API shape intentional and stable).

## Assumptions

- Symbols table is already migrated and seeded (`AAPL`, `MSFT`, `SPY`).

## Failure Modes

- Missing env load can route Prisma to wrong DB or no DB; fixed by loading dotenv in runtime.
- If migrations are not applied to target Neon branch, endpoint returns DB table-not-found errors.

## How to Test (commands + steps)

```bash
cd apps/api
npm run start
```

Then:

```bash
curl http://localhost:3001/symbols
```

Expected:
- HTTP 200
- Array of symbol objects with `id`, `ticker`, `name`, `createdAt`, `updatedAt`
