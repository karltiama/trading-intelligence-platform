## Context

The API already had Phase 0 Nest modules (`health`, `market-data`, `signals`, `indicators`, `scheduler`). We needed to introduce Prisma cleanly for Neon/Postgres without adding trading logic yet.

## Goal

Set up Prisma in `apps/api`, define MVP models and relations, add Nest Prisma integration, and keep the app passing build/lint/tests.

## Files Changed

- `apps/api/prisma/schema.prisma`
- `apps/api/prisma.config.ts` (generated by Prisma init)
- `apps/api/src/prisma/prisma.service.ts`
- `apps/api/src/prisma/prisma.module.ts`
- `apps/api/src/app.module.ts`
- `apps/api/.env.example`
- `apps/api/package.json` / `apps/api/package-lock.json` (dependencies)

## Implementation Notes (key logic explained)

- Prisma initialized under `apps/api/prisma` with PostgreSQL provider.
- Phase 0 models added: `Symbol`, `Candle`, `Signal`, `StrategyRun`.
- Added enums for data consistency: `Timeframe`, `SignalSide`, `RunStatus`.
- `PrismaService` uses Prisma 7 Postgres adapter (`@prisma/adapter-pg` + `pg`) and Nest lifecycle hooks.
- `PrismaModule` is global and exports `PrismaService` for feature modules.
- `PrismaModule` imported into `AppModule`.

## Key Decisions (and alternatives considered)

- **Use Decimal DB columns** for price/volume fields to avoid float drift in trading data.
- **Use Prisma adapter pattern** (Prisma 7 compatible) instead of legacy constructor assumptions.
- **No repository layer yet** in this step to stay focused on Phase 0 scaffolding.

## Assumptions

- Real Neon connection string will be provided in local `apps/api/.env`.
- Migrations are run from `apps/api`.

## Failure Modes

- Missing/invalid `DATABASE_URL` will block migration commands.
- Neon IP/network restrictions can fail migration even with correct URL.

## How to Test (commands + steps)

```bash
cd apps/api
npx prisma format
npx prisma generate
npm run build
npm run lint
npm run test
npm run test:e2e
```
