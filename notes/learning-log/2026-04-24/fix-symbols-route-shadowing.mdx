## Context

`GET /symbols` returned a legacy payload because an older route in `AppController` shadowed the new symbols module route.

## Goal

Ensure `GET /symbols` is served only by `SymbolsController` so tracked-symbol fields (`isActive`) are returned.

## Files Changed

- `apps/api/src/app.controller.ts`
- `apps/api/src/app.service.ts`

## Implementation Notes (key logic explained)

- Removed legacy `@Get('symbols')` from `AppController`.
- Removed unused Prisma-based `listSymbols()` from `AppService`.
- Kept `AppController` focused on root health/hello behavior.

## Key Decisions (and alternatives considered)

- **Decision:** remove legacy route entirely.
  - **Why:** prevents ambiguous route ownership and inconsistent response shapes.
  - **Alternative:** keep route and proxy to `SymbolsService` (unnecessary indirection).

## Assumptions

- `SymbolsController` is now the canonical owner for symbol management routes.
- No external client depends on the legacy `/symbols` response shape.

## Failure Modes

- If another duplicate `/symbols` route is added later, route conflicts can reappear.
- If `SymbolsModule` is removed from `AppModule`, `/symbols` will 404.

## How to Test (commands + steps)

1. `npm run build:api`
2. `npm run test:api`
3. Start API: `npm run dev:api`
4. Call `GET /symbols` and verify each row includes `isActive`.
5. Call `PATCH /symbols/NVDA/toggle` then `GET /symbols` to verify state change.

## Next Steps

- Add an e2e test that asserts `/symbols` response shape includes `isActive`.
- Add route ownership notes in module docs to avoid duplicate paths.
